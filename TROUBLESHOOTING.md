# TROUBLESHOOTING · 踩坑排查与工程取舍记录

> 本文档记录在单卡 RTX 3090 上从零搭建 Paper2Code 过程中遇到的真实环境与工程问题、排查思路、解决方案，以及受限条件下的关键取舍。
> 重点不在"踩了多少坑"，而在每个问题的**根因分析**和**工程判断**。

---

## 一、环境与依赖类

### 1.1 vLLM FlashInfer CUDA 算子即时编译失败

**现象**：较新版本 vLLM 启动时报 `Engine core initialization failed`，深层日志显示 FlashInfer 的 `sampling.cu` 编译出现上百个错误，核心是 `cub::BlockAdjacentDifference` 的 `FlagHeads` 成员不存在。

**根因**：FlashInfer 是 vLLM 用于加速采样的库，首次运行会即时编译（JIT）CUDA 算子。新版 FlashInfer 内部依赖的 cub 库接口与本机 CUDA 工具链版本不匹配——某些类成员在新版 cub 中已移除/改名，导致编译失败。本质是 vLLM、FlashInfer、本机 CUDA 三者的版本耦合冲突。

**解决**：按"改动从小到大"依次评估——切换 attention backend 绕过该算子、卸载 FlashInfer 让 vLLM fallback、降级 vLLM 到稳定版本。最终通过降级到 vLLM 0.6.3（配套 torch 2.4 + CUDA 12.1）解决。

**经验**：受限硬件 + 部署场景下，**版本的稳定性优先于新颖性**。新版本对底层工具链要求更高，在老架构（Ampere）上容易触发编译问题。

### 1.2 版本降级引发的依赖污染连锁

**现象**：降级 vLLM 后接连报 `libcudart.so.13 cannot open shared object file`（找不到 CUDA 13 运行时库）、`No module named 'pyairports'` 等看似无关的错误。

**根因**：同一环境里先装高版本（CUDA 13 时代的包），后降级到低版本（CUDA 12 配套），导致新旧依赖混杂。残留的 torchaudio 是为高版本 torch 编译的，仍找 CUDA 13 的库；outlines 的子依赖在降级中丢失。典型的"环境污染"——修好一个又冒下一个。

**解决**：零散缺包逐个补齐效率低且不收敛，最终**重建干净环境**：删除被污染环境，新建后只装目标版本及配套依赖一条线。

**经验**：**一个项目一个隔离环境**，不要在装过高版本的环境里降级。零散修复超过两三次仍不收敛时，果断重建比继续修更省时间——这是工程上的"止损"判断。

### 1.3 离线/内网环境的模型下载问题

**现象**：直连 HuggingFace 下载模型超时；同样命令在 A 终端能下、B 终端 `Network is unreachable`。

**根因**：两层。一是网络层面需走国内镜像（hf-mirror）；二是 `HF_ENDPOINT` 等环境变量**不跨终端共享**——只在设置它的终端生效，新终端没有该变量就去连官方源失败。

**解决**：使用 hf-mirror 镜像；将镜像配置、conda 环境激活等写入 shell 配置文件，对所有新终端默认生效。

**经验**：环境变量、conda 激活、端口等都是**会话级上下文，不跨终端继承**。把固定初始化沉淀到配置文件，避免"同样命令换终端就报错"的困惑。

### 1.4 检索库与推理环境的依赖互斥（关键取舍）

**现象**：在 vLLM 环境装 Docling、FlagEmbedding（BGE-M3）时连续冲突：
- `cannot import name 'AutoModelForImageTextToText' from transformers`（Docling 需新版 transformers，但环境是迁就 vLLM 的旧版）
- `XLMRobertaModel.__init__() got an unexpected keyword argument 'dtype'`（新版 FlagEmbedding + 旧版 transformers）
- `module 'pyarrow' has no attribute 'json_'`（新版 datasets + 旧版 pyarrow）

**根因**：**依赖夹心**。vLLM 0.6.3 要求 transformers 不能太新（上限），而 Docling/FlagEmbedding 要求 transformers 足够新（下限），两个区间不重叠 → 无解。在单一环境里硬扛必然冲突。

**解决（核心架构决策）**：物理隔离为两个独立 conda 环境——主环境 `paper2code` 跑 vLLM（旧 transformers），新环境 `paper2code-rag` 跑 Docling + BGE-M3 + Qdrant（新 transformers）。两环境通过网络接口（后续用 FastAPI/MCP）和共享 Qdrant 通信。

**经验**：这本质是**微服务解耦思想**——模型推理服务和数据处理服务本就该独立管理依赖。被依赖冲突逼出来的解耦，反而成了项目的架构亮点。"无法在一个环境里调和的依赖，用独立环境隔离"是通用解法。

### 1.5 Docling 的 OCR 组件版本不兼容

**现象**：独立环境装好 Docling 后，解析时报 `ValueError: Unsupported configuration: torch.PP-OCRv6.det.small`。

**根因**：Docling 默认加载 OCR 模型（RapidOCR）用于识别图片文字，但请求的 OCR 模型配置版本不被支持，是 Docling 与 rapidocr 的版本小不匹配。

**解决**：识别到测试论文是文字版 PDF（非扫描件），**根本不需要 OCR**。在 pipeline 配置里 `do_ocr = False` 关闭 OCR，既绕开版本坑又更快。

**经验**：报错时先判断"这个功能我是否真的需要"。关掉一个用不上的组件，比去修它更省事。

## 二、服务部署类

### 2.1 端口占用与进程残留

**现象**：启动 vLLM 报 `Address already in use`，但 `lsof` 查端口却显示为空。

**根因**：vLLM 启动会拉起多个子进程（API server、engine、worker）。异常退出时主进程结束，子进程可能残留并仍占端口；且 TCP 端口释放有 TIME_WAIT 延迟，立即重启会再冲突。

**解决**：`pkill -9 -f vllm` 按进程名彻底清理残留，等端口释放后重启；或换端口（同步更新客户端配置）。

**经验**：多进程服务清理不能只 kill 主进程。`lsof`/`ss`/`fuser`/`ps` 多工具交叉验证更可靠。

### 2.2 服务能启动但请求时 500：懒加载暴露的缺包

**现象**：vLLM 正常启动，但发请求时返回 500，服务端报 `No module named 'pyairports'`。

**根因**：guided decoding 相关库（outlines）是懒加载——只有收到需要它的请求时才 import。所以启动阶段一切正常，缺包问题延迟到处理请求时才暴露。

**经验**：服务"能启动"不等于"功能完整"。懒加载会让依赖问题延迟到运行时，**验证要覆盖真实业务请求**，不能只看启动成功。

### 2.3 Docker 镜像拉取失败

**现象**：`docker build` 报 `docker.mirrors.ustc.edu.cn: no such host` / Docker Hub 连接超时。

**根因**：配置的镜像加速器（中科大源）域名已失效；Docker Hub 官方源国内直连不稳定。

**解决**：更换为当前可用的镜像加速器后 build 成功。若加速器均不可用，则保留 Dockerfile 作为容器化方案、说明构建思路，不在网络问题上死磕。

**经验**：Docker 镜像加速器在国内不稳定。`docker build` 失败若纯属网络环境问题，**Dockerfile 本身就是容器化能力的证据**，不必为跑通一个 build 耗费过多时间。

### 2.4 容器网络与残留容器

**现象**：`docker run` 报 `address already in use`（8080 被宿主机 uvicorn 占用）、`container name already in use`（上次失败的容器未删）。

**解决**：`docker rm -f <name>` 清理残留容器；清掉占用端口的进程；改用 `--network host` 让容器共享宿主机网络，使容器内能通过 localhost 访问宿主机的 vLLM(8001)/MCP(8000) 服务。

**经验**：容器有独立网络，容器内 `localhost` 指容器自己而非宿主机。容器要访问宿主机服务时，`--network host` 是单机场景下最简单的解法。

## 三、硬件能力边界与取舍

### 3.1 FP8 KV cache 在 3090 上不可用

**问题**：计划用 FP8 量化 KV cache 省显存，但 RTX 3090 是 Ampere 架构，**FP8 需要 Ada Lovelace / Hopper 架构（4090、H100）的硬件原生支持**，是芯片层面的能力边界，软件无法绕过。

**取舍**：放弃 FP8，改用 3090 真正支持的组合——FP16 KV cache + PagedAttention（提升利用率）+ prefix caching（复用 system prompt）+ NF4 权重量化。

**经验**：技术选型必须匹配硬件能力。**了解一项优化的硬件前提，和会用它同样重要**。坚持只写在当前硬件上真实做到的优化，避免面试中细节对不上。

### 3.2 模型尺寸与显存的现实匹配

**问题**：原计划部署约 30B 级模型，但单卡 24GB 即便 NF4 量化（权重约 16-18GB），留给 KV cache 和激活的空间极紧，易 OOM 或只能开极短上下文。

**显存账（NF4，单卡 24GB）**：框架预留+激活约 3-4GB，可用约 20GB。30B 级权重 16-18GB → KV cache 仅剩 2-4GB；14B 级权重 8-9GB → KV cache 充裕。

**取舍**：确立"主攻 14B（稳定）、有余力冲更大尺寸（加分）、小尺寸兜底"的分级策略，最终稳定部署 Qwen2.5-14B（NF4）。

**经验**：**先算账再下载几十 GB 的模型**。基于显存预算做容量规划，避免"下载半天才发现放不下"。

### 3.3 模型选型的事实核查

**问题**：最初计划的模型，其标称版本号和参数尺寸在官方实际发布中并不存在。

**解决**：核查官方信息后，改用确实存在、架构成熟、对当前偏旧推理环境兼容性好的 Qwen2.5 系列。

**经验**：技术信息以官方一手来源为准，不凭印象。选型除了看能力，还要看与现有环境（推理框架版本、是否多模态架构）的兼容性。

## 四、评测方法的迭代

### 4.1 绝对打分区分度不足

**现象**：用 LLM-as-Judge 对完整 Agent 和 naive RAG baseline 各自打绝对分（1-5），结果分数高度集中（多为 3.5），两者几乎分不开，甚至 baseline 略高。

**根因**：LLM 当裁判做绝对打分有"居中偏好"，倾向给安全的中间分，对两份都"还行"的报告难以区分细微差异。

**解决**：改用 **pairwise 对比评测**——让裁判同时看两份报告、直接判断哪个更好，相对比较比绝对打分敏感得多。并加入**位置偏见消除**：两个顺序各评一次（A在前/B在前），结论一致才计，否则算平局。

**结果**：改进后得到可信结论——Agent 2 胜 0 负 4 平，相比 baseline 未现劣势、部分样本更优。

**经验**：评测方法本身也需要验证和迭代。发现"评测测不出差异"时，先判断是被评对象真没差异，还是评测方法不敏感。pairwise + 位置偏见消除是 LLM 评测的常用可靠做法。

### 4.2 评测的已知局限（诚实记录）

- **自评偏差**：受本地资源限制，裁判与被评者用同一模型，存在偏差；理想应用更强的独立裁判模型。
- **评测集偏小且偏经典**：论文多为方法清晰的经典论文，Agent 架构发挥空间有限；其优势预期在复杂、信息分散的任务上更明显。评测集应按难度分层扩充。

**经验**：主动记录评测的局限，比给出一个无法解释的漂亮数字更专业。诚实的负面/温和结果 + 到位的分析，胜过夸大的结论。

## 五、整体经验提炼

1. **稳定性优先于新颖性**——受限硬件 + 部署场景，验证过的稳定版本组合远比最新版省心。
2. **环境隔离是底线**——一项目一环境；依赖互斥时按职责拆分独立环境；修复不收敛时果断重建。
3. **会算账再动手**——显存、磁盘等容量规划前置，避免无效下载和反复试错。
4. **优化要匹配硬件**——理解每项优化的硬件前提（如 FP8 的架构要求），只用真正能用的。
5. **诚实自洽**——实际做了什么就记录什么，让每个技术细节经得起追问。
6. **系统化排查**——从最深层报错入手（而非笼统的收尾报错）、多工具交叉验证、按"改动从小到大"依次尝试。
